Common Terraform Scenarios

Scenario: Duplicate Resources Problem
Problem: When using local state and GitHub only, if Engineer A creates an EC2 instance but forgets to push the updated state file, Engineer B might pull the repo without the latest state and create a duplicate EC2 instance.

Solution: Use remote state with S3 backend. This ensures all team members work with the same state file, preventing duplicate resources.
Best Practice: GitHub + S3 Workflow
Step 1: Store Terraform code (.tf files) in GitHub
Step 2: Configure S3 backend for state storage
Step 3: Run terraform init to connect to remote backend
Step 4: Make changes and run terraform apply
Step 5: State file is automatically updated in S3
Recovery: Use S3 versioning to restore previous state if needed

Variables and Outputs in Terraform

What is a Variable?

A variable in Terraform is like a placeholder. It lets you store a value (like a region name, server type, etc.) so you can reuse it in your code.

What is an Output?

An output or output variable shows useful information after you run terraform apply. It's like Terraform saying: "Here's the result you might want to see!"

Variable Example:

variable "region" { default = "us-east-1" }

Output Example:

output "bucket_name" { value = aws_s3_bucket.my_bucket.bucket }

Why Use Outputs?

  • To see values like IP addresses, bucket names, or resource IDs
  • To pass values between modules
  • To debug and check values

Multi-Cloud vs Multi-Region

Multicloud in Terraform

What it means: Using more than one cloud provider in the same Terraform project.

Example: You deploy a web app on AWS and a database on Azure.

Why it's used: To avoid putting all your services on one provider (vendor lock-in), or to take advantage of features from different providers.

Multiregion in Terraform

What it means: Deploying resources in multiple regions of the same cloud provider (like AWS).

Example: Deploying servers in us-east-1 and us-west-1 to improve performance or reliability.

Why it's used: For disaster recovery, performance, or compliance.

Multicloud Provider Example:

provider "aws" { region = "us-east-1" } provider "azurerm" { features = {} }

Multiregion Provider Example:

provider "aws" { alias = "us_east" region = "us-east-1" } provider "aws" { alias = "us_west" region = "us-west-1" } resource "aws_instance" "east_server" { provider = aws.us_east # ... configuration } resource "aws_instance" "west_server" { provider = aws.us_west # ... configuration }

Terraform Modules

Terraform Module Structure

What is a Module in Terraform?

A module in Terraform is a folder that contains Terraform code and can be reused in different places by passing values to it. Think of it like a Lego block - you build it once, you can plug it into multiple places, and you can pass values to it while it gives outputs back.

Why use Modules?

  • To organize your code
  • To reuse the same setup (e.g., EC2 instance, VPC) in multiple environments
  • To avoid copy-pasting the same code everywhere

Module Structure

  • main.tf - Contains resources
  • variables.tf - Defines input variables
  • outputs.tf - Defines output values

Data Flow

Value from main.tf → goes into → variable block in module → used in → resource or logic inside module's main.tf

Using a Module:

module "ec2_instance" { source = "./modules/ec2_instance" ami_value = "ami-0530dd4ac4c456" instance_type_value = "t2.micro" subnet_id_value = "subnet-0130e91ed99b52e" }

Terraform State Management

Local vs Remote State Management

What is a Terraform State File?

When you use Terraform to create cloud resources (like servers or databases), it needs to remember what it created. That's what the state file does! It's a file called terraform.tfstate that keeps track of all the resources Terraform manages.

Problems with Local State

  • Security Risk: State file may contain secrets like passwords, API keys
  • No Locking: Multiple people can't work safely at the same time
  • Duplicate Resources: Risk of creating duplicate infrastructure

Benefits of Remote State (S3)

  • Secure & Encrypted: S3 provides encryption and IAM controls
  • State Locking: DynamoDB prevents concurrent modifications
  • Versioning: Rollback to previous state versions

Remote Backend Configuration:

terraform { backend "s3" { bucket = "my-terraform-state-bucket" key = "terraform.tfstate" region = "us-east-1" dynamodb_table = "terraform-locks" encrypt = true } }

Workflow with Remote State:

1. Code in GitHub

Store Terraform code (.tf files) in GitHub repository

2. State in S3

State file is securely stored in S3 with encryption

3. Locking with DynamoDB

DynamoDB handles state locking for team collaboration

4. Automatic Updates

State is automatically updated in S3 after each apply

Advanced Terraform Concepts

Provisioners

In Terraform, provisioning means setting up or configuring a resource after it's created. It's like saying: "Hey Terraform, after you create this server, run this script to install software or make some changes inside it."

file Provisioner

Uploads a file to the server

remote-exec Provisioner

Runs commands on the server via SSH

local-exec Provisioner

Runs a command on your local machine

Provisioner Example:

resource "aws_instance" "example" { ami = "ami-123456" instance_type = "t2.micro" provisioner "remote-exec" { inline = [ "sudo apt update", "sudo apt install -y nginx" ] } }

Terraform Workspaces

Terraform workspaces let you use the same code for different environments (like dev, stage, prod) by creating a separate state file for each one.

terraform workspace list # Shows all workspaces terraform workspace new dev # Creates new workspace terraform workspace select dev # Switches to workspace terraform workspace show # Shows current workspace

Terraform with HashiCorp Vault

Vault is a tool to securely store secrets like passwords, API keys, and access tokens. Instead of putting secrets directly in your Terraform code (which is risky), you can store secrets in Vault and configure Terraform to read those secrets.

Step-by-Step Process:

1. Secrets Stored in Vault

Example: Store db_password = "supersecret" in Vault at path like secret/data/db

2. Terraform Authenticates with Vault

Terraform uses an AppRole, Token, or other auth method. Vault checks if Terraform is allowed to access the secret.

3. Terraform Reads Secret

Terraform uses the Vault provider to fetch the secret. The secret is used during provisioning (like setting up a database).

4. No Secrets in Code!

The sensitive data stays in Vault. Terraform just reads it at runtime.
